# Sprint 1 — Demand Insight Technical Stories

These stories describe technical capabilities implemented and validated through
Day 21. They do not represent backend API or React dashboard functionality.

## TS-S1-001 — Establish the sales data contract and loader

### Necesidad técnica

The system needs a stable sales schema and a loader that rejects missing, empty
or unsupported inputs before downstream processing.

### Criterios de aceptación

- The required raw columns are documented.
- An existing non-empty CSV can be loaded as a DataFrame.
- Missing files, empty datasets and missing required columns produce explicit failures.

### Estado

Completed.

### Evidencia

- `docs/data-contract.md`
- `ai-services/demand-insight/src/data/data_loader.py`
- `ai-services/demand-insight/checks/check_data_loading.py`
- `tests/ai-services/demand-insight/test_data_processing.py`

### Relación con User Stories

US-S1-001, US-S1-006.

## TS-S1-002 — Clean and validate retail sales

### Necesidad técnica

The system needs a deterministic cleaning flow that normalizes types and removes
records that cannot support reliable sales analysis.

### Criterios de aceptación

- Required columns are validated before cleaning.
- Dates and numeric fields are converted safely.
- Missing values, duplicates and negative units or prices are removed.
- The official clean dataset contains 18 valid rows and can be generated reproducibly.

## TS-S1-004 — Engineer temporal and revenue features

### Necesidad técnica

The system needs reusable transformations that derive temporal signals and
observed economic value without changing the number of sales records.

### Criterios de aceptación

- Valid dates produce day-of-week, month, year and weekend indicators.
- Revenue equals `units_sold * unit_price`.
- Feature engineering preserves row count and exposes expected numeric/boolean types.
- Invalid dates or numeric inputs fail explicitly.

### Estado

Completed.

### Evidencia

## TS-S1-005 — Calculate the mean baseline and MAE

### Necesidad técnica

The system needs a simple reference prediction and an interpretable error metric
before introducing any future modeling work.

### Criterios de aceptación

- The baseline equals the mean of valid `units_sold` values.
- Prediction length matches target length.
- MAE is correct, non-negative and rejects incompatible lengths.
- The official dataset produces a baseline of 16.28 and MAE of 5.42.

### Estado

Completed.

## TS-S1-009 — Separate production, automated tests and manual verification

### Necesidad técnica

The module needs explicit responsibility boundaries so production, automated
verification, manual evidence and operational execution do not duplicate logic.

### Criterios de aceptación

- `src/` owns production logic and official artifact generation.
- `tests/` contains isolated pytest coverage and writes only to temporary paths.
- `checks/` provides readable end-to-end verification with non-zero failure behavior.
- `reports/` stores evidence and `scripts/` provides repeatable execution.
- The suite reports 30 passing tests and every discovered manual check passes.

### Estado

Completed.

- `tests/ai-services/demand-insight/`

## TS-S1-011 — Assemble the versioned Demand Summary

### Necesidad técnica

The platform needs one internal read service that converts validated analytical
artifacts into a stable product response before HTTP and React are introduced.

### Criterios de aceptación

- The service validates every required CSV and JSON artifact before responding.
- The response exposes schema version, period, sales summary, baseline, leaders,
  five Insight Cards and explicit limitations.
- Missing, empty or structurally invalid evidence raises `DemandSummaryError`.
- Tests use a temporary project root and never write official output paths.
- The operational script generates the official JSON deterministically.

### Estado

Completed.

## TS-S1-012 — Expose Demand Summary through a strict HTTP contract

### Necesidad técnica

The platform needs a stable HTTP boundary so a frontend can consume the Demand
Summary without accessing analytical files or internal exception details.

### Criterios de aceptación

- `GET /api/v1/demand-insights/summary` returns schema version `1.0` and the
  structured service response.
- Strict Pydantic schemas reject missing, invalid or additional response fields.
- `DemandSummaryError` is mapped to a generic `503 Service Unavailable` response.
- Health and OpenAPI documents are available.
- Endpoint tests isolate the service through dependency overrides.
- Runtime and transitive dependencies are pinned.

### Estado

Completed.

### Evidencia

- `backend/api/app/main.py`
- `backend/api/app/routes/demand_summary.py`
- `backend/api/app/schemas/demand_summary.py`
- `tests/backend/test_demand_summary_api.py`
- `backend/api/checks/check_demand_summary_api.py`
- `backend/api/requirements-lock.txt`
